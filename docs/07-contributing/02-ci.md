# CI and Testing Guide

**Audience:** contributors working on the registry codebase. For operator-facing deployment notes, see [reference/configuration.md](../05-reference/03-configuration.md) and [operations/ops.md](../06-operations/01-ops.md).

All gates the project enforces are `make` targets defined in the root `Makefile`. CI platforms wire to those targets — they do not redefine the commands. This means the same gates run identically in local dev, in pre-commit, and in any CI runner.

---

## Quick reference

```bash
make all                # everything a PR must pass (lint + format-check + typecheck + doc-refs + test-hygiene + test)
make lint               # ruff check — style + correctness lints
make format-check       # ruff format --check — formatting gate (read-only)
make format             # ruff format — apply formatting (writes files)
make typecheck          # mypy --strict across source roots
make doc-refs           # no internal-doc references in shipped code
make test-hygiene       # no phase-named test files or stale phase comments
make test-unit          # unit tests — pure Python, no DB, ~2s for 1100+ tests
make test-integration   # integration tests — testcontainers Postgres, slow
make test-conformance   # conformance suite — OpenAPI drift, tenant isolation, MCP catalog
make test-perf          # SLO perf tests — latency p95 verification, marked @slow
make test               # unit + conformance (fast gates for daily dev)
```

---

## Gate descriptions

### lint (`make lint`)

Runs `ruff check` across `registry/registry`, `registry/sync`, and `registry/scripts`. Catches style violations, unused imports, and common correctness issues. Required on every PR.

### format-check (`make format-check`)

Runs `ruff format --check` — fails if any file needs reformatting. Use `make format` (writes files) to fix locally before pushing. Required on every PR.

### typecheck (`make typecheck`)

Runs `mypy --strict` on the full source tree. The project ships no `type: ignore` overrides in new code; if mypy reports an error it is a real problem. Required on every PR.

### doc-refs (`make doc-refs`)

Verifies no shipped file contains references to internal planning documents (ADR numbers, PRD feature IDs, task IDs, TDD section citations, phase labels). The gate walks `registry/registry`, `registry/sync`, `registry/scripts`, and `registry/docs`. Exit non-zero with a `file:line` list on any hit.

The rule: code comments explain the reason in the code's own vocabulary, not where the rule was decided. See the repo-root `CLAUDE.md` for the full list of forbidden patterns and the intentional-bypass mechanism.

### test-hygiene (`make test-hygiene`)

Verifies no test file under `registry/tests/` is named with a delivery-phase label and no test comment contains a stale phase marker. The gate is `registry/scripts/check_no_phase_named_tests.py`. If a test legitimately uses "phase" as a domain term (not a delivery milestone), end the relevant line with `# test-hygiene: intentional`.

### test-unit (`make test-unit`)

Runs all tests under `registry/tests/unit/`. Unit tests are pure Python with no database dependency. They use SQL-string-keyed `AsyncMock` routers to fake the DB layer. Target: ~1.5–2 seconds for 1100+ tests. This is the default home for new tests.

### test-integration (`make test-integration`)

Runs `registry/tests/integration/` with a 180-second timeout. Uses testcontainers to spin up a real Postgres instance. Promotes to integration when the unit version cannot exercise the real code path (triggers, constraints, partitioning, cross-service behaviour).

### test-conformance (`make test-conformance`)

Runs `registry/tests/conformance/` with a 60-second timeout. These are contract-drift gates:

- **OpenAPI snapshot** — verifies the committed `openapi.json` matches the live schema.
- **MCP tool catalog** — verifies the registered MCP tools match the expected list.
- **Cross-tenant isolation invariants** — verifies every query path funnels through the visibility chokepoint.

Run conformance before tagging a release.

### test-perf (`make test-perf`)

Runs `registry/tests/perf/` with a 300-second timeout. Tests are marked `@pytest.mark.perf @pytest.mark.slow` and excluded from `make test` (the daily fast gate). Run in the release pipeline to verify latency p95 SLOs and webhook fan-out timing.

## Test pyramid

Pick the right bucket when adding a test:

| Bucket | When to use | Speed |
|---|---|---|
| `tests/unit/` | Default. Pure Python, no DB. Use SQL-string-keyed `AsyncMock` to fake the DB layer. | ~1.5s for 1100+ |
| `tests/integration/` | Only when behaviour spans more than one service or requires real SQL (triggers, constraints, partitioning). | Slow; needs Docker |
| `tests/conformance/` | Contract drift gates. Run before tagging a release. | Moderate; needs Docker |
| `tests/perf/` | SLO verification. Run in release pipeline only. | Slow; marked `@slow` |

When in doubt, write a unit test first. Promote to integration only when the unit version cannot exercise the real code path.

## Pre-commit hooks

```bash
pre-commit install
```

After installation, every `git commit` runs lint, format-check, typecheck, doc-refs, and test-hygiene automatically. No manual invocation needed. The same targets that CI runs are what pre-commit runs — they share the `Makefile` targets, not separate configs.

## CI platform wiring

Two example wirings ship with the project:

- `registry/.github/workflows/` — GitHub Actions
- `registry/.gitlab-ci.yml` — GitLab CI

Both call the `make` targets above. Neither is required — you can delete either without affecting the gates. Any CI runner (Jenkins, Buildkite, CircleCI, Azure DevOps, Bitbucket Pipelines, a plain `bash` script) that can invoke `make` can wire these gates.

A typical PR pipeline runs:

```
make all      →  lint + format-check + typecheck + doc-refs + test-hygiene + test-unit + test-conformance
```

A release pipeline adds:

```
make test-integration
make test-perf
make openapi-export   (verify openapi.json is committed and current)
make build-docker
```

## Environment for tests

Unit tests need no environment variables — the fake DB layer requires no database connection.

Integration and conformance tests need Docker available. The testcontainer framework starts and stops Postgres automatically; no pre-running Postgres is required.

If you need to point integration tests at an external Postgres (e.g., in a CI environment without Docker access), set `DATABASE_URL` before running `make test-integration`. The testcontainer framework skips its own Postgres when `DATABASE_URL` is already set.

---

## Regenerating the OpenAPI snapshot

The committed `openapi.json` must match the live schema. Regenerate it with:

```bash
make openapi-export
```

Then commit the updated file. The conformance suite (`make test-conformance`) fails if the committed file is stale.

---

## Adding new Make targets

If you add a new gate, add it to the `Makefile` at the repo root under the appropriate section. Wire it into `make all` if it must pass on every PR. Keep each target as a single command or short pipeline — no logic in Make beyond invocation. Document what it does in the `## <description>` inline comment that `make help` prints.
